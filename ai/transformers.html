Transformers — How LLMs Actually Read and Think

Complete architecture deep dive from tokens to KV cache. How attention works, why Q/K/V exist, what causal masking does, and every production trade-off you need for MAANG IC4 interviews — with real code and real failure modes.

01What is a Transformer?

A Transformer is a neural network architecture designed to process sequences. In LLM work the sequence is usually text — but Transformers can also process code, images, audio, video, tables, and tool calls. They are the foundation of every major language model: GPT-4, Claude, LLaMA, Gemini, Mistral.

The central idea is deceptively simple: instead of processing tokens one at a time (like older RNN-based models), a Transformer lets every token look at every other token simultaneously and update its own meaning based on the full context around it.

The Context Problem

Consider these two sentences:

"The trophy did not fit in the suitcase because it was too large." → "it" = trophy "The trophy did not fit in the suitcase because it was too small." → "it" = suitcase

Same word, same position, completely different meaning. A Transformer resolves this by building a contextual representation of each token — a vector whose value is informed by every token surrounding it. This is the whole problem the architecture was designed to solve.

Use in interviews

"A Transformer turns tokens into contextual representations using alternating attention and MLP layers. Attention lets tokens communicate; MLP computes within each token."

02Why Transformers Replaced RNNs

Before Transformers (pre-2017), sequence modeling used RNNs, GRUs, and LSTMs — models that processed tokens one at a time from left to right, maintaining a hidden state that carried information forward. Three deep structural problems drove the transition.

Problem 1 — Long-range Dependencies

In the sentence "The contract that the legal team reviewed last Friday was approved", the word "was" needs to agree with "contract" — not with "team" or "Friday", which are closer. An RNN must carry the information about "contract" through every intermediate step without losing it. In practice, gradient flow degrades across many steps, and important early information fades.

A Transformer eliminates this entirely. In the attention mechanism, "was" can directly attend to "contract" in one step regardless of distance. The connection is explicit and learned.

Problem 2 — Poor Parallelism

RNN token 5 depends on the output of token 4, which depends on token 3. This chain means training is fundamentally sequential — you cannot compute position 5 until you have computed positions 1 through 4. This made it impossible to utilise modern GPU parallelism effectively.

Transformers process all token relationships in parallel during training. This single change is why it became possible to train on datasets of trillions of tokens — and why GPT-3 scale models exist at all.

Problem 3 — Fixed Bottleneck

Seq2seq RNN architectures (used for translation before Transformers) compressed the entire input sequence into a single fixed-size hidden vector, which the decoder then tried to reconstruct the meaning from. No matter how long the input, it had to fit in one vector. Transformers replaced this with rich many-to-many token-level connections.

03The End-to-End Pipeline

The model never processes raw text. Every stage transforms the input into a progressively richer numerical representation. Here is the complete path from string to predicted token for a decoder-only LLM:

pipeline # Input: raw text prompt = "The capital of France is" # Step 1: Tokenization token_ids = tokenizer.encode(prompt) # [791, 6864, 315, 9822, 374] # "The" "capital" "of" "France" "is" # Step 2: Embedding lookup + position encoding token_embeddings = embedding_table[token_ids] # shape: [5, d_model] pos_embeddings = position_encoding(positions) # shape: [5, d_model] x = token_embeddings + pos_embeddings # Step 3: Pass through N Transformer blocks for block in transformer_blocks: # 32 blocks in GPT-2, 96 in GPT-3 x = block(x) # each block: Attention + MLP + Residuals + Norm # Step 4: Output head logits = linear_head(x[-1]) # shape: [vocab_size] ~50,000 values next_token = sample(logits) # "Paris"

The model's final answer emerges from the last token's hidden state in a decoder-only LLM. That last position aggregates information from all previous positions via attention — which is why the quality of the context window matters so much.

04Tokenization

The model never sees characters or whole words directly. A tokenizer splits text into sub-word units called tokens and maps each to an integer ID from a fixed vocabulary.

# Sub-word tokenization (BPE / WordPiece) "unbelievable" → ["un", "believ", "able"] → [1342, 4492, 902] "The" → 791 "capital" → 6864 "of" → 315 "France" → 9822 "is" → 374 # Vocabulary sizes # GPT-2: 50,257 tokens # LLaMA: 32,000 tokens # GPT-4: ~100,000 tokens # Rule of thumb: # 1 token ≈ 3/4 of an English word # 128k token context ≈ 200 pages of text

Sub-word tokenization is a principled compromise: common words like "the" become a single token; rare words like "unbelievable" are split into frequent sub-units. This keeps vocabulary size manageable while handling any string including code, URLs, and non-English text.

Context windows are measured in tokens, not words or characters. When estimating if your document fits, use ~1.3 tokens per English word as your multiplier. Code is denser — often closer to 1 token per 3–4 characters.

05Embeddings & Contextual Representations

Each token ID maps to a dense vector called an embedding — a row in a learned matrix (the embedding table) with one row per vocabulary token. These initial vectors capture coarse semantic properties learned during training.

Related tokens end up geometrically close: king, queen, monarch cluster near each other; SQL, table, query, database cluster near each other. This geometric structure is useful and emerges purely from predicting the next word on large text corpora.

# The same word gets DIFFERENT representations after Transformer layers sentence_1 = "I opened a bank account yesterday" sentence_2 = "We sat by the bank of the river" sentence_3 = "The bank shot off the cushion" # In all three: "bank" starts with the same embedding vector # After 32 Transformer layers: # sentence_1: "bank" → [financial institution, account, deposit...] # sentence_2: "bank" → [riverbank, geography, water, terrain...] # sentence_3: "bank" → [billiards, shot, cushion, angle...] # This is a CONTEXTUAL embedding — the whole point of deep Transformers

The embedding lookup gives a token a context-free starting vector. After all Transformer layers, that vector becomes fully context-aware. Building those contextual representations layer by layer is the entire job of the Transformer blocks between input and output head.

06The Transformer Block

A Transformer model is a stack of identical blocks. Each block has two major sublayers that alternate. Understanding one block is understanding 80% of how the whole model works.

The block in compact form (pre-norm style)
x = x + Attention(Norm(x))
x = x + MLP(Norm(x))
Pre-norm (normalise before sublayer) is used by GPT, LLaMA, Mistral and most modern LLMs — more stable than post-norm for deep networks

Attention Layer

Tokens communicate with each other. Each token updates its representation by gathering information from other tokens in the sequence, weighted by learned relevance. After this step, each token's vector reflects not just its own meaning but the meaning of everything around it.

MLP / Feed-Forward Layer

Each token is transformed independently. After attention has gathered context from other tokens, the MLP processes each token's representation on its own — applying nonlinear transformations that refine and enrich the representation. MLP layers hold a large share of the model's parameters and are where much factual knowledge is stored. Do not treat them as minor components.

# MLP structure inside a block def mlp(x): x = linear(x, in_dim=4096, out_dim=16384) # expand to 4x x = gelu(x) # nonlinearity x = linear(x, in_dim=16384, out_dim=4096) # project back return x # Residual connection (applied around every sublayer) x = x + mlp(norm(x)) # output = input + change, not replacement

Residual Connections

Every sublayer adds its output to its input rather than replacing it: x = x + sublayer(x). This creates a gradient highway through deep networks — the original signal flows straight through and each layer learns only the refinement needed. Without residuals, training 96-layer networks like GPT-3 would be nearly impossible due to vanishing gradients.

Layer Normalization

Applied before each sublayer (pre-norm style), LayerNorm or RMSNorm stabilises the scale of activations, preventing them from growing too large or too small during the forward pass. Modern LLMs predominantly use RMSNorm (a simpler variant) because it is faster and numerically stable.

The line to memorise

"Attention = communication between tokens. MLP = computation inside each token." Use this exact sentence in every interview. It is the most concise correct description of the architecture.

07Self-Attention — Tokens Looking at Each Other

Self-attention is the mechanism inside the attention sublayer that lets each token update its representation by looking at other tokens in the same sequence. Crucially, the attention pattern is dynamic — it changes with every input sentence rather than being fixed by position like in a CNN.

# Sentence 1: "it" attends heavily to "animal" "The animal did not cross the street because it was tired." # Attention weights for "it": # animal: 0.62 tired: 0.24 street: 0.08 others: 0.06 # Sentence 2: same model, same architecture, different pattern "The animal did not cross the street because it was crowded." # Attention weights for "it": # street: 0.62 animal: 0.08 crowded: 0.20 others: 0.10 # The model learned that crowded streets → "it" refers to the street # No rule was hard-coded. This emerged from next-token prediction training.

Self-attention enables Transformers to outperform RNNs at tasks requiring long-range understanding because the connection between any two tokens is always one attention step, regardless of their distance in the sequence.

08Q, K, V — The Mechanism

Every token produces three vectors by multiplying its current representation by three separate learned weight matrices. These are the Query, Key, and Value — the core of the attention mechanism.

Computing Q, K, V from a token embedding x
Q = x · WQ    K = x · WK    V = x · WV
WQ, WK, WV are learned matrices of shape [d_model, d_head] — trained end-to-end by backpropagation

The Mental Model

  • Query (Q) — "What am I looking for?" For the pronoun "it": "I need to find the noun I refer to."
  • Key (K) — "What do I contain / advertise?" For "animal": "I'm a living noun, a possible referent." For "street": "I'm a place noun, also a possible referent."
  • Value (V) — "What information can I pass along if attended to?" The actual semantic content of that token.

Think of it like a database lookup: Q is the search query, K is the document index, V is the document content. You compare the query against the index to find relevant documents, then retrieve the content from those documents.

# For token "it" in "The animal did not cross the street because it was tired" # Query of "it": # "looking for: subject noun that can be tired" # Keys of other tokens: # "animal" key: living noun, animate, possible subject → HIGH match with Q # "street" key: inanimate place noun → moderate match # "tired" key: adjective, applies to animate things → useful context # "cross" key: verb, action → low match # Attention scores (before softmax): scores = Q_it @ K_all.T # dot product: [1, seq_len] weights = softmax(scores / sqrt(d_k)) # New representation of "it": new_it = sum(weights[i] * V[i] for i in range(seq_len)) # = 0.62 * V(animal) + 0.24 * V(tired) + 0.08 * V(street) + ...

Q and K are what determine where to look. V is what actually gets mixed in. This separation means you can have very different routing logic (Q·K) from information content (V) — a key design strength.

09The Attention Formula — Explained Term by Term

The full attention formula looks intimidating written as one expression. Each term has a clear job. You do not need to derive it — but you must be able to explain every part fluently.

Scaled Dot-Product Attention
Attention(Q, K, V) = softmax( Q·KT / √dk ) · V
dk = dimension of the key/query vectors (e.g., 64 for a 64-dimensional head)

Q·KT — compute relevance scores

The matrix product compares every Query vector against every Key vector simultaneously. If token i has a Query that is similar in direction to token j's Key, they get a high raw score. This is the "who should I pay attention to?" step.

÷ √dk — scale to prevent saturation

Without scaling, dot products between high-dimensional vectors tend to grow large in magnitude, pushing softmax into saturation regions where gradients are near zero. Dividing by √dk keeps the pre-softmax scores in a reasonable range regardless of dimension size.

softmax — normalise to a probability distribution

Turns the raw scores into weights that sum to 1.0. The highest-scoring token gets the most weight; very low-scoring tokens get near-zero weight. This is what "attention" literally means — where does each token direct its attention?

· V — weighted sum of value vectors

The final step produces the output: a weighted combination of Value vectors. Each output token representation is a mixture of information from all input tokens, blended according to the learned attention weights.

Interview delivery

"QK-transpose computes relevance scores between all token pairs; divide by root-d prevents saturation; softmax converts to a probability distribution; multiply by V produces the context-aware output." Say this exactly. It is the complete correct answer in four clauses.

10Multi-Head Attention

A single attention head gives one view of the relationships in the sequence. Language carries many simultaneous relationship types at once — subject-verb agreement, pronoun resolution, local syntax, long-range dependency, code bracket matching, tool-call schema tracking. A single head cannot specialise in all of these simultaneously.

Multi-head attention solves this by running several attention operations in parallel, each in its own learned subspace:

multi-head attention def multi_head_attention(x, num_heads=8): head_dim = d_model // num_heads # e.g., 4096 / 8 = 512 per head outputs = [] for h in range(num_heads): # Each head has its own W_Q, W_K, W_V (different learned subspace) Q = x @ W_Q[h] # [seq, head_dim] K = x @ W_K[h] # [seq, head_dim] V = x @ W_V[h] # [seq, head_dim] attn = softmax(Q @ K.T / sqrt(head_dim)) @ V outputs.append(attn) # Concatenate all heads, project back to d_model combined = concat(outputs, dim=-1) # [seq, d_model] return combined @ W_O # output projection # GPT-3: 96 layers, 96 heads, d_model=12288, head_dim=128

What Different Heads Learn

Research has found that individual heads can develop interpretable roles: some track subject-verb links, some resolve pronoun references, some specialise in local syntax, some handle long-range dependencies. However, this is not guaranteed — many heads have no clean human-readable interpretation.

Do not overstate head interpretability in interviews. Say: "Different heads can learn to attend to different relationship types, though not every head has a clean human-interpretable role. What matters is that multiple subspaces give the model richer collective representations than any single attention operation."

11Positional Encoding

Self-attention is permutation-invariant by default. If you shuffle the tokens in a sequence, the attention scores are identical (just rearranged). The model treats the input as a set, not a sequence.

# Same tokens, opposite meaning — pure attention can't distinguish these "The dog bit the man." # dog = aggressor "The man bit the dog." # man = aggressor # Without position info, both sentences have identical attention matrices # (just with rows/columns permuted)

Position encodings fix this by adding a position-dependent vector to each token embedding before the first Transformer block:

# input to Transformer = token embedding + positional encoding x[i] = token_embed[token_id[i]] + pos_encode[i] # After this, "dog" at position 1 has a DIFFERENT vector than "dog" at position 3 # The model can now distinguish sequence order

Position Methods

MethodUsed byKey property
SinusoidalOriginal 2017 TransformerFixed formula using sin/cos at varying frequencies. Can generalise to sequences longer than training length.
Learned embeddingsGPT-2, BERTTrain a position vector for each absolute position. Simple but cannot extend beyond training length.
RoPE (Rotary)LLaMA, Mistral, modern LLMsEncodes relative position by rotating Q and K vectors. Generalises better to longer contexts. Current standard.
ALiBiSome open-source modelsAdds a position-based bias directly to attention logits. No positional embeddings at all.

RoPE is the dominant choice in modern LLMs because it naturally encodes relative position (how far apart two tokens are) rather than absolute position (where token is in absolute terms). This makes it more robust to context length extension and long-document tasks.

"Encoder-only models are best for understanding tasks because they can condition on both left and right context. 13Causal Masking

For decoder-only models trained with next-token prediction, the model must not see future tokens during training — or it would trivially learn to copy the answer rather than predict it. Causal masking enforces this constraint and also ensures that training exactly matches inference behaviour.

# Training text: "I love machine learning" # The model is trained on all positions in parallel using the causal mask # Position 0: sees ["I"] → predicts "love" # Position 1: sees ["I", "love"] → predicts "machine" # Position 2: sees ["I", "love", "machine"] → predicts "learning" # Implemented as a lower-triangular mask on the attention score matrix: mask = tril(ones(seq_len, seq_len)) # 1 = attend, 0 = block scores = scores.masked_fill(mask == 0, float('-inf')) weights = softmax(scores) # -inf → 0 after softmax # This single operation lets training process all positions at once # while preventing any position from "seeing" its future

At inference time, generation is sequential — one new token at a time, each only seeing the tokens generated so far. The causal mask ensures training and inference are perfectly aligned: the model never saw future tokens during training, so it has learned to predict without them.

14Context Window & Attention Cost

The context window is the maximum number of tokens the model can process in a single forward pass. The model can only attend to information within this window — anything outside it is invisible to the model.

# Context window sizes in the wild # GPT-3.5: 4,096 tokens # GPT-4: 32,768 tokens (extended to 128k) # Claude 3: 200,000 tokens # Gemini 1.5: 1,000,000 tokens

The O(n²) Attention Problem

Standard self-attention compares every token against every other token. For a sequence of n tokens, the attention matrix has n² entries. Memory and compute both scale quadratically.

# Attention matrix sizes n = 1_000 → 1_000_000 token-pairs # fine n = 10_000 → 100_000_000 token-pairs # expensive n = 100_000 → 10_000_000_000 pairs # GPU memory bottleneck # This is why long-context models are expensive to serve # and why O(n) or O(n log n) attention variants are an active research area

The "Lost in the Middle" Failure

A larger context window does not guarantee the model uses all information within it. Research has consistently shown that LLMs pay less attention to information buried in the middle of very long prompts — they are better at using information near the start and end. This is called the "lost in the middle" problem.

Do not assume that putting more text in the context window automatically improves performance. For large knowledge bases, targeted RAG retrieval often outperforms blindly expanding context because it delivers the relevant text to the front of the prompt where the model attends to it most reliably.

15KV Cache — Making Generation Practical

During inference, a decoder-only LLM generates one token at a time. Without caching, generating each new token would require recomputing the Key and Value vectors for every previously generated token. That is O(n) recomputation per step and O(n²) total — prohibitively wasteful.

KV cache — before and after # WITHOUT KV cache (naive) # Generating "Paris" requires processing the full sequence each step: step_1: process ["The", "capital", "of"] → "France" step_2: process ["The", "capital", "of", "France"] → "is" step_3: process ["The", ... "is"] → "Paris" # wasteful! # WITH KV cache step_1: compute K,V for ["The", "capital", "of"] → cache them → "France" step_2: compute K,V for ["France"] only → reuse cached K,V → "is" step_3: compute K,V for ["is"] only → reuse cached K,V → "Paris" # Only the new token's K,V is ever computed from scratch # O(1) per step instead of O(n) per step

The Memory Cost

KV cache trades compute for memory. Cache size grows with every token generated and is proportional to: sequence length × number of layers × number of heads × hidden dimension × 2 (K and V). For long sequences with large batch sizes on large models, the KV cache becomes the GPU memory bottleneck.

16Training Objective & The Hallucination Problem

Next-Token Prediction

GPT-style decoder-only models are trained on a single objective: given all previous tokens, predict the next one. This is computed in one parallel forward pass using the causal mask to generate predictions at all positions simultaneously.

# Training on the sentence: "The cat sat on the mat." # All predictions computed in a single forward pass: input: ["The", "cat", "sat", "on", "the"] targets: ["cat", "sat", "on", "the", "mat"] loss = cross_entropy(logits, targets) # averaged over all positions # This single objective, applied to trillions of tokens, teaches: # - Grammar and syntax # - Factual knowledge # - Reasoning patterns # - Code structure # - World knowledge # But NOT: truthfulness, safety, instruction following, alignment

Why Hallucinations Are Structural, Not Bugs

An LLM is trained to generate plausible continuations, not verified truths. When the model encounters a prompt asking for information it doesn't have grounded evidence for, it generates whatever text pattern has the highest probability given the context — which can look exactly like a correct answer while being completely wrong.

# The hallucination mechanism query = "Give me the citation for the Smith 2019 butterfly migration study." # Model has no memory of a real citation. It asks itself: # "What is the most plausible next token sequence given this query?" # Answer: something that looks like a citation. output = "Smith, J. et al. (2019). Monarch butterfly migration patterns. " "Nature, 571(7), 234-239." # Fluent, well-formatted, correct structure, entirely fabricated.

Mitigation Strategies

  • RAG — retrieve relevant source documents and include them in context; model answers from retrieved evidence rather than parametric memory
  • Tool use — let the model call APIs, databases, or search engines for factual information instead of generating it from training weights
  • Citations — require the model to cite specific sources; hallucinated claims are harder to fabricate if citations must be verifiable
  • Abstention training — fine-tune the model to output "I don't know" when confidence is low, rather than confabulating
  • Structured outputs — schema validation catches formatting errors and forces the model to stay within a constrained answer space
  • Evaluation pipelines — RAGAS, LLM-as-judge, human eval; you cannot improve what you don't measure

17Sampling & Decoding

Once the model produces logits (a score for every token in the vocabulary), a sampling strategy selects the next token. This controls the trade-off between consistency and creativity.

# After the forward pass, the model outputs logits over vocab logits = model(prompt)[-1] # shape: [vocab_size] e.g. 50,000 # Convert to probabilities probs = softmax(logits / temperature) # temperature scales the distribution # temperature = 0.0 → greedy (always pick highest prob token) # temperature = 0.2 → near-greedy, very consistent # temperature = 0.7 → balanced (default for most chat models) # temperature = 1.2 → more diverse, more creative, more errors # Top-k: sample only from the k most likely tokens top_k_probs = top_k(probs, k=50) # ignore the long tail # Top-p (nucleus): sample from smallest set summing to probability p top_p_probs = top_p(probs, p=0.9) # adaptive cutoff based on distribution next_token = sample(top_p_probs)
SettingEffectBest for
Temperature 0.0Deterministic — always picks highest-prob tokenJSON extraction, tool calls, structured output
Temperature 0.2–0.4Near-deterministic, very consistentFactual Q&A, code generation
Temperature 0.7Balanced consistency and diversityChat, instruction following (common default)
Temperature 0.9–1.2High diversity, more creative, more errorsBrainstorming, creative writing, idea generation

18Production Trade-offs

For an IC4 AI Platform role, architectural knowledge must connect to operational reality. Every Transformer deployment involves three-way trade-offs between latency, cost, and quality.

Latency Sources

Two phases dominate LLM inference latency. Prefill (processing the input prompt) scales with prompt length — longer prompts take longer. Decode (generating output tokens) is sequential and scales with output length. KV cache size and GPU memory bandwidth constrain how many requests can be batched simultaneously.

Debugging Wrong Outputs

When an LLM application produces a wrong answer, do not immediately reach for fine-tuning. Work through the chain first:

  1. Was the user query understood correctly?
  2. Was the right context retrieved (if using RAG)?
  3. Was that context included in the prompt, or was it too long / noisy?
  4. Did the model ignore the relevant evidence in context?
  5. Was the output schema validated?
  6. Was sampling temperature too high?
  7. Is the model too weak for this task complexity?
  8. Do existing evals capture this failure mode?
  9. Do we need a guardrail, fallback, or human escalation?

Large Model vs Small Model

DimensionLarger modelSmaller model
Reasoning qualityBetterWeaker on complex tasks
Instruction followingMore reliableMay need fine-tuning
LatencyHigherLower
Cost per tokenHigherLower (often 10-100x)
DeployabilityNeeds large GPU clusterCan run on single GPU or CPU

The production pattern: route simple, high-volume requests to small fast models; route complex, low-volume requests requiring nuanced reasoning to large models. This dramatically reduces cost while preserving quality where it matters. Build routing logic before defaulting everything to the largest available model.

Core Architecture

  • A Transformer = stacked blocks of Attention + MLP + Residuals + Norm
  • Attention = communication between tokens; MLP = computation inside each token
  • Every token produces Q, K, V by multiplying embedding by three learned matrices
  • Attention formula: softmax(QK⊃T / √d) · V
  • Residual connections (x = x + sublayer(x)) are what makes deep networks trainable
  • Pre-norm (normalise before sublayer) is the modern standard

Key Mechanisms

  • Self-attention is dynamic — the pattern changes with every input sentence
  • Multi-head attention runs several attention operations in parallel subspaces
  • Positional encoding is required because attention is permutation-invariant by default
  • RoPE encodes relative position via Q/K rotation — standard in modern LLMs
  • Causal masking blocks future tokens — makes training match inference
  • KV cache stores past K,V; only new token computed each decode step

Model Types

  • Encoder-only (BERT): bidirectional attention, best for understanding tasks
  • Decoder-only (GPT, LLaMA): causal attention, best for generation
  • Encoder-Decoder (T5, BART): source → encoder, output → decoder with cross-attn
  • Most production LLMs are decoder-only: simpler, scales well
  • Use encoder-only for retrieval embeddings in RAG systems

Production Realities

  • Standard attention is O(n²) — long contexts are expensive
  • "Lost in the middle" — models attend less to information in long prompt middles
  • Hallucinations are structural: model predicts plausible text, not verified facts
  • Mitigate with RAG, tool use, citations, abstention, evals
  • Debug wrong outputs step by step before reaching for fine-tuning
  • Route simple requests to small models — cost can drop 10–100x

🎯 Interview Questions — with answers

Click each question to expand the answer. These are the questions most likely to come up for IC4-level AI engineering roles.

Q1 Explain Transformer architecture. Walk me through how an input prompt becomes a predicted token. +
The prompt is first tokenised into integer IDs, then looked up in an embedding table to produce dense vectors. Positional encodings are added to give the model order information. The sequence then passes through N Transformer blocks, each consisting of a self-attention sublayer and a feed-forward MLP sublayer, both wrapped with residual connections and layer normalisation. Self-attention lets each token gather context from all others by computing Query, Key, and Value vectors and taking a weighted combination of values according to Query-Key relevance. The MLP then transforms each token independently. After N blocks, the last token's hidden state is projected to logits over the vocabulary by a linear output head, and sampling produces the next token. For decoder-only models, a causal mask prevents each position from attending to future positions during both training and generation.
Q2 Explain Q, K, V without equations. Why do we need three separate vectors instead of one? +
Each token produces three vectors by multiplying its embedding by three separate learned matrices. The Query represents what that token is looking for — for a pronoun like "it," the query encodes "I need to find the noun I refer to." The Key represents what each token advertises or contains — "animal" might signal "living noun, potential referent." The Value is the actual information passed along if a token is attended to. The model compares Queries with Keys to produce relevance scores, normalises them with softmax, then blends the Values by those weights. The reason for three separate vectors rather than one is the separation of concerns: Q and K control routing (who attends to whom) while V controls content (what information flows). This lets the model learn routing patterns independently from information content, which is a powerful design choice.
Q3 Why does multi-head attention exist? What do different heads actually learn? +
A single attention head produces one view of the sequence — one set of relevance weights. But a sentence simultaneously contains multiple relationship types: subject-verb agreement, pronoun resolution, local syntax, long-range dependency, and for code, bracket matching or function call tracking. A single head must compromise across all of these. Multi-head attention runs H attention operations in parallel, each with its own Q/K/V weight matrices projecting into a different subspace, and concatenates the outputs. Research has found that individual heads can specialise — some track syntactic structure, some handle coreference, some attend to positionally adjacent tokens. However, I would not overstate interpretability in interviews: many heads have no clean human-readable role. What matters is that multiple subspaces give the model richer collective representations than any single attention operation could produce.
Q4 What is causal masking and why is it essential for both training and inference? +
Causal masking prevents each token from attending to future tokens by setting the attention score for any future position to negative infinity before softmax, making the weight effectively zero. It is essential for two separate reasons. During training: if the model could see the next token it is trying to predict, it would trivially learn to copy it rather than building real language understanding. The mask forces the model to predict each position using only past context. During inference: the model only ever has access to previously generated tokens, so training must match this constraint exactly. If training used bidirectional attention but inference is causal, the model would encounter inputs it was never trained on. Causal masking also enables a training efficiency trick: all positions in the sequence can be trained in one parallel forward pass, rather than one pass per position.
Q5 What is KV cache and what are its trade-offs at scale? +
KV cache stores the Key and Value vectors for all previously generated tokens during autoregressive generation. Without it, generating each new token would require recomputing K and V for the entire sequence so far — O(n) per step, O(n²) total. With KV cache, only the new token's K and V are computed; past K, V are read from cache. This reduces per-step compute to O(1) in terms of past tokens, enabling the fast generation speeds users experience. The trade-offs at scale are significant: cache size grows as sequence length × layers × heads × hidden dimension × 2, and must be maintained separately per request in the batch. At large batch sizes with long contexts, the KV cache becomes the GPU memory bottleneck rather than compute. Production systems address this with techniques like prefix caching (share cache for identical system prompts across requests) and grouped query attention (fewer unique K,V heads to reduce cache size).
Q6 Why do LLMs hallucinate and what are the practical mitigations? +
Hallucinations are a structural property of how LLMs are trained, not an engineering bug you can simply fix. The model is trained to predict the most plausible next token given the context — not the most truthful one. When the model lacks grounded evidence in its training data or context, it still generates fluent text because that is what it was optimised to do. It cannot distinguish "I don't know, but this sounds right" from "I know this is correct." Practical mitigations work by either providing grounding (RAG retrieves relevant source documents so the model answers from evidence rather than memory; tool use fetches real-time facts from APIs or databases), constraining outputs (structured schemas, citation requirements, abstention training to say "I don't know"), or catching failures (evaluation pipelines that measure factuality, human review for high-stakes domains, monitoring for retrieval quality in RAG). The key production insight is that no single mitigation is sufficient — production systems combine grounding, constraints, and evaluation.
Q7 What is the difference between encoder-only and decoder-only models, and how does that determine what tasks each is suited for? +
The critical difference is the attention mask. Encoder-only models (BERT, RoBERTa) use bidirectional attention — every token can attend to every other token, past and future. This makes them excellent at understanding tasks because the representation of each token is informed by its full surrounding context. They are the right choice for classification, named entity recognition, semantic embeddings for retrieval, and extractive question answering. They cannot naturally generate open-ended text because there is no causal structure. Decoder-only models (GPT, LLaMA, Claude) use causal attention — each token can only see past tokens. They are trained with next-token prediction, which teaches them to generate coherent continuations. This makes them the right choice for text and code generation, chat, instruction following, tool calling, and agentic tasks. The training objective and the masking strategy are deeply linked — you cannot trivially apply next-token prediction training to a bidirectional model because it would cheat by looking at the answer.
Q8 How would you reduce latency for an LLM application in production? +
I would approach it at four levels. First, model selection: route simple, high-volume requests to smaller/distilled models — this is often the highest-impact change and can reduce cost and latency 10–100x. Second, input length: prompt compression, summarising long conversation histories, removing unnecessary context. Third, output length: setting lower max_tokens, requesting structured data instead of prose where possible, streaming tokens to improve perceived latency even when total generation time is unchanged. Fourth, infrastructure: enable KV prefix caching for shared system prompts (amortises prefill cost across requests), batching concurrent requests, using speculative decoding for common patterns, and quantisation to reduce memory bandwidth pressure. Before optimising, I would profile to find the actual bottleneck — whether it is prefill, decode, retrieval latency, or tool call round-trips, because each requires a different fix.
