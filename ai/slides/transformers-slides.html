Slide 2 of 14 — What is a Transformer
The Core Idea

Every token looks at every other token

A Transformer is a neural network that processes sequences. Unlike older models that read left-to-right, a Transformer lets every token attend to every other token simultaneously — updating its meaning based on full context.

Attention = communication between tokens. MLP = computation inside each token. These two operations alternate in every Transformer block.

SAME WORD, DIFFERENT MEANING
"...too large" → it = trophy
"...too small" → it = suitcase
03 / 14
Slide 3 of 14 — Why Transformers Won
Before Transformers

Why RNNs and LSTMs couldn't scale

Problem 1

Long-range dependencies. RNNs carry information step-by-step — by step 50, early info has faded. Transformers connect any two tokens directly.

Problem 2

Poor parallelism. RNN token 5 depends on token 4's output — training is sequential. Transformers process all relationships in parallel → scales to GPT-size.

Problem 3

Fixed bottleneck. Seq2seq RNNs compressed the whole input into one hidden vector. Transformers use rich many-to-many token connections.

04 / 14
Slide 4 of 14 — Pipeline
End-to-End

Text → tokens → vectors → prediction

The model never processes raw words. Every stage transforms the input into a richer numerical representation until a probability distribution over vocabulary emerges at the end.

Tokenizer — splits text into sub-word units, maps to integer IDs
Embedding — each ID → dense vector. Add position info.
Blocks × N — attention + MLP refine representations layer by layer
Output head — linear → logits → sampling → next token

1 token ≈ ¾ English word. 128k token context ≈ 200 pages. GPT-4 vocab ≈ 100,000 tokens.

Slide 5 of 14 — The Transformer Block
The Building Block

Stack of attention + MLP

Compact form
x = x + Attention(Norm(x))
x = x + MLP(Norm(x))
Pre-norm style used by GPT, LLaMA, Mistral

Attention layer

Tokens communicate. Each token updates itself by gathering information from other relevant tokens.

MLP layer

Tokens compute. Each token transformed independently. Holds most of the model's factual knowledge.

Residual connections (the + signs) are the gradient highway that makes 96-layer networks like GPT-3 trainable. Each layer refines — it doesn't replace.

Slide 6 of 14 — Self-Attention
Self-Attention

Each token asks: who should I look at?

The answer changes with every sentence. Self-attention is dynamic — the model learns which tokens matter to each other from the data, not from fixed rules.

WHY THIS MATTERS

Notice how clicking "it" in sentence 1 shows high attention to "animal" — but in sentence 2 it shifts to "street." Same word, same model, different context-aware result.

WHAT ATTENTION COMPUTES

compare → score → weight → combine

1. Compare my query with all keys
2. Score how relevant each token is
3. Softmax → weights sum to 1
4. Weighted sum of values → my new repr.

This is why Transformers replaced RNNs. Direct token-to-token connections, learned dynamically, in parallel.

Slide 7 of 14 — Q, K, V
The Mechanism

Query · Key · Value

Every token produces three vectors by multiplying its embedding by three separate learned weight matrices: W_Q, W_K, W_V.

Q — Query: "What am I looking for?"

"it" (pronoun) → looking for its referent noun

K — Key: "What do I contain?"

"animal" → I'm a living noun, possible referent
"street" → I'm a place noun, also possible referent

V — Value: "What info can I share?"

Semantic information about that token — passed along weighted by attention score

Think: Q = search query · K = document index · V = document content

Slide 8 of 14 — Attention Math
The Formula

You don't derive it. You explain it.

Scaled Dot-Product Attention
softmax( Q·Kᵀ / √dk ) · V
Q·Kᵀ — relevance scores
Dot product of query with every key. High score = "these tokens are relevant to each other."
÷ √d — scale factor
Prevents scores from becoming too large, which would push softmax into saturation (near-zero gradients).
softmax — normalize to weights
Turns raw scores into a probability distribution. All weights sum to 1.0.
· V — weighted sum
Blend value vectors by relevance weights. Output = context-aware representation.

ATTENTION WEIGHTS FOR TOKEN "it"

Sentence: "…animal…street…it was tired"
animal
0.62
tired
0.24
street
0.08
others
0.06
new repr. of "it" ≈ 0.62·V(animal) + 0.24·V(tired) + …

"QK-transpose computes relevance scores; divide by root-d prevents saturation; softmax converts to weights; multiply by V produces the context-aware output." — use this exact sentence in interviews.

Slide 9 of 14 — Multi-head & Positional
Multi-head Attention

Multiple relationship views in parallel

One attention head gives one view of the sequence. Language carries many simultaneous relationship types — multi-head attention runs several attention operations in parallel, each in its own learned subspace.

Head 1 (example)
subject–verb links
Head 2 (example)
pronoun → referent
Head 3 (example)
local syntax
Head N …
long-range deps

⚠ Many heads have no clean human-readable interpretation — multiple subspaces ≠ labelled functions.

Positional Encoding

Attention is order-agnostic by default

"The dog bit the man" and "The man bit the dog" are the same set of tokens. Without position info, the model treats them identically.

FIX: add position vectors to embeddings

input = token_embed + position_embed
Sinusoidal — original 2017 paper, fixed formula
Learned — GPT-style, train a vector per position
RoPE — LLaMA, Mistral, modern LLMs. Encodes relative position via Q/K rotation. Better generalisation to longer sequences.

"Transformers need positional encoding because self-attention is order-agnostic by default. Position encodings tell the model where each token sits in the sequence."

Slide 10 of 14 — Encoder vs Decoder
Architecture Variants

Same block. Different masking.

🔍

Encoder-only

BERT · RoBERTa · DeBERTa

Bidirectional attention — each token sees left AND right context simultaneously. Best for understanding.

✓ Classification
✓ Semantic embeddings
✓ Retrieval / reranking
✓ NER, extractive QA
✗ Open-ended generation
✍️

Decoder-only

GPT · LLaMA · Mistral · Claude

Causal attention — each token only sees past tokens. Trained with next-token prediction. Best for generation.

✓ Text & code generation
✓ Chat, instruction following
✓ Tool calling / agents
✓ Reasoning, planning
🔄

Encoder-Decoder

T5 · BART · translation models

Encoder reads input (bidir), decoder generates output via cross-attention. Best for structured input→output.

✓ Translation
✓ Summarization
✓ Structured I/O tasks

Most modern LLMs (GPT-4, Claude, LLaMA) are decoder-only. The decoder-only architecture is simpler, scales well, and emergent capabilities appeared at large scale with next-token prediction alone.

Slide 11 of 14 — Causal Masking
Decoder-only Detail

Causal masking: no peeking at future tokens

Training with next-token prediction requires that the model not see the token it's predicting. Causal masking enforces this — and ensures training exactly matches inference.

TRAINING (parallel)

All positions computed in one forward pass. The causal mask blocks each position from seeing future ones. Efficient — processes the full sequence at once.

INFERENCE (sequential)

One token at a time. Each new token only sees previously generated tokens. Training matches inference — that's why it works.

Blocked positions get score = −∞ before softmax. After softmax, weight → 0. Future tokens are invisible.

Slide 12 of 14 — Context & KV Cache
Context Window

Bigger context ≠ always better

The context window is the max tokens the model can attend to. Standard attention scales O(n²) — every token compares with every other token.

1k tokens → 1,000,000 comparisons
10k tokens → 100,000,000 comparisons
100k tokens → 10B comparisons
GAINS
More info available · fewer retrieval failures
COSTS
↑ latency · ↑ memory · ↑ cost · "lost in the middle"
KV Cache

Store past K, V. Don't recompute.

WITHOUT CACHE — naïve

Generating token 5 → recompute K,V for tokens 1–4
Generating token 6 → recompute K,V for tokens 1–5
O(n²) total compute — wasteful

WITH KV CACHE ✓

Store K,V for past tokens once. For each new token, compute only its own Q, K, V — attend to cached past.
O(n) per step → fast generation
Memory cost grows with: sequence length × layers × heads × hidden dim
Bottleneck at large batch size on GPU

KV cache trades compute for memory. It's why throughput on long-context requests is memory-bound, not compute-bound.

Slide 13 of 14 — Production
Production Reality

Why hallucinations happen — and what to do

Hallucinations — the mechanism

LLMs are trained to generate plausible text, not verified truth. When context is missing or ambiguous, the model fills in convincingly but incorrectly.

"Give me the citation for this." → model has no source → fabricates a plausible-looking reference.

✓ Fix: RAG · tool use · citations · abstention training · structured outputs · evals

Debugging wrong answers

Don't jump to fine-tuning. Check the chain first:

1. Was the query understood correctly?
2. Was the right context retrieved?
3. Was context in the prompt, or too noisy?
4. Did the model ignore relevant evidence?
5. Was output schema validated?
6. Was sampling too random?
7. Is the model too weak for this task?

LATENCY

Model size · input/output length · KV cache pressure · tool calls · retrieval

COST

Input + output tokens · GPU-hours · storage · reranking

RELIABILITY

Structured outputs · retries · fallback · RAG · evals · human review

SAFETY

Prompt injection · tool abuse · PII leakage · over-permissioned agents
