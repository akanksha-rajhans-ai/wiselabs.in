Semantic Search — Interview Cheat Sheet

AI Engineering Day 1 ⚡ Premium

Key Terms — One-liner definitions
BM25
Lexical ranking algorithm combining term frequency and inverse document frequency. Fast, explainable, blind to synonyms and paraphrases.
Semantic Search
Retrieval by meaning rather than exact word match. Query and documents are embedded into vectors; nearest vectors are returned as results.
Cosine Similarity
Similarity metric measuring the angle between two vectors, not their magnitude. Score: 1.0 (identical) to −1.0 (opposite). Standard for text embeddings.
ANN Search
Approximate Nearest Neighbour — finds close vectors without scanning everything. Trades ~5% recall for 100–1000x speed vs exact search.
HNSW
Hierarchical Navigable Small World — the most common ANN algorithm. Builds a layered graph; navigates greedily from sparse to dense layers at query time.
Hybrid Search
Combining BM25 and semantic search results. BM25 handles exact matches; semantic handles meaning. Merged using Reciprocal Rank Fusion.
RRF (Reciprocal Rank Fusion)
Algorithm to merge multiple ranked lists using rank position (not raw scores). score = Σ 1/(k + rank). Default k=60. No normalisation needed.
Bi-encoder
Embeds query and document independently into vectors. Fast, scalable. Used for first-stage retrieval. Limitation: no cross-attention between query and doc.
Cross-encoder
Takes query and document as a pair, outputs a relevance score. Slow (can't pre-compute). Highly accurate. Used for second-stage reranking only.
Two-Stage Retrieval
Retrieve 50 candidates fast with bi-encoder → rerank to top 5 with cross-encoder. Standard production pattern for high-precision retrieval systems.
Formulas Worth Knowing
Cosine Similarity
cos(θ) = (A · B) / (‖A‖ × ‖B‖)
When vectors are L2-normalised (as most models output), this equals dot product. 1.0 = same meaning · 0.0 = unrelated · −1.0 = opposite.
Reciprocal Rank Fusion
score(d) = Σ 1 / (k + rank(d))
k=60 (default, empirically chosen). Sum over all rankers. Uses rank positions not raw scores — no normalisation needed across incompatible scales.
BM25 (simplified)
IDF(q) · f(q,D) · (k₁+1) / (f(q,D) + k₁)
k₁ ≈ 1.2 (TF saturation) · b ≈ 0.75 (length normalisation). Backbone of Elasticsearch, Lucene, Solr.
HNSW Key Params
M · ef_construction · ef_search
M = connections per node (default 16). ef_construction = index quality (default 200). ef_search = query recall vs speed tradeoff.

Interview Questions — 3-sentence answers
Q1Walk me through what happens when a user submits a query in a semantic search system.
The query is passed through the same embedding model used during indexing, producing a query vector. This vector is sent to the vector database, which uses an ANN algorithm like HNSW to return the top-K most similar document vectors by cosine similarity. Optionally, these candidates are passed through a cross-encoder reranker for precision before being returned or fed to an LLM.
Q2Why combine keyword search and semantic search in production?
Pure semantic search fails on exact string matches — product codes, error messages, proper nouns — where the nearest semantic neighbour might be completely wrong. Pure keyword search misses synonyms, paraphrases, and intent. Hybrid search with RRF gets the best of both: BM25 handles structured exact matches, semantic handles natural language, and they're merged without needing to normalise incompatible score scales.
Q3What is RRF and why k=60?
RRF merges multiple ranked lists by assigning each document score = 1/(k + rank) per list and summing — using rank positions, not raw scores, so no normalisation is needed. The constant k=60 was found empirically to reduce the outsized influence of rank-1 results while still rewarding consistently high-ranked documents. It's robust and parameter-free in practice — just use the default.
Q4What are the failure modes of semantic search?
Four main ones: exact string needs (serial numbers, error codes where nearest semantic neighbour is wrong), out-of-domain vocabulary (model wasn't trained on the terminology — embeddings are noise), very short queries (no context for the model — "Python" is ambiguous), and stale embeddings (document updated but not re-indexed — silent failure). Each has a different fix: hybrid search, domain fine-tuning, query expansion, and a re-indexing pipeline respectively.
Q5How does ANN differ from exact nearest neighbour search and why does it matter?
Exact NN is O(n·d) per query — at 10M vectors with 768 dimensions, that's billions of multiplications, far too slow for interactive latency. ANN algorithms like HNSW build a navigable graph structure allowing greedy search in O(log n), trading ~5% recall for 100–1000x speed gains. In practice the 5% you miss are usually lower-quality results — ANN is always the right choice at production scale.
Q6If semantic search quality is poor, what would you investigate first?
Start with chunk quality — bad chunking (too small, cut mid-sentence) is the most common cause and is invisible without manually inspecting the chunks. Then check raw cosine scores — if everything is below 0.5, the query is likely out-of-domain. Finally verify model consistency — query and index must use the same model, and a mismatch produces garbage results silently. For systematic diagnosis, run RAGAS context precision and recall to pinpoint where in the pipeline retrieval is losing information.
Say This If Stuck
If asked about choosing between BM25 and semantic search
"In production I'd default to hybrid — BM25 handles exact matches and structured data well, semantic handles natural language and intent. The performance of either alone is almost always worse than both combined with RRF, so unless there's a strong reason to simplify, hybrid is my starting point."
If asked about poor retrieval quality
"The most likely culprit is chunking — it's the part most people get wrong and it has an outsized impact on retrieval quality. I'd inspect the chunks manually first before touching the model or the retrieval algorithm."
If asked why cosine similarity instead of Euclidean distance
"Cosine measures the angle between vectors, not their length — so two sentences meaning the same thing point in the same direction regardless of how long they are. Euclidean distance would penalise longer sentences even when the meaning is identical, which is the wrong signal for text."
If asked about model selection for embeddings
"For local development and learning I'd use all-MiniLM-L6-v2 — free, fast, runs locally. For production, I'd evaluate text-embedding-3-small from OpenAI — excellent quality-to-cost ratio. If I need the best open-source option, BGE-large from BAAI consistently tops the MTEB benchmark."
Common Mistakes Interviewers Catch
✗
Using different embedding models for indexing and querying — produces completely wrong results silently. ✓ Always use the same model. If you change models, re-embed the entire corpus.
✗
Treating cosine similarity scores as probabilities ("85% confident this is relevant"). ✓ Cosine scores are not calibrated probabilities. Never present them as such without explicit calibration.
✗
Using semantic search for exact identifiers (order IDs, error codes, serial numbers). ✓ Use BM25 or direct lookup for structured identifiers — semantic nearest neighbour is wrong here.
✗
Skipping reranking for high-stakes top-K results and relying on ANN output directly. ✓ ANN is approximate. For precision-critical results, add a cross-encoder reranker on top-50 candidates.
✗
Forgetting that document updates require re-embedding — assuming the index is always fresh. ✓ Always have a re-indexing pipeline triggered on document update events.
