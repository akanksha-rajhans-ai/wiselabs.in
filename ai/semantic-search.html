Semantic Search — Finding Meaning, Not Words

Why keyword search fails in modern AI products and how embedding-based retrieval fixes it. The engine behind every serious search and RAG system — with real code, real failure modes, and real production patterns.

01The Problem With Keyword Search

For decades, search engines worked by matching words. You search for "broken phone screen", the engine looks for documents containing those exact tokens. This is called lexical search — and the algorithm that powers most of it is called BM25.

How BM25 Actually Works

BM25 (Best Match 25) scores a document for a query by combining two signals: how often a term appears in the document (term frequency), and how rare that term is across all documents (inverse document frequency). Rare terms that appear frequently in a document score high.

BM25 Scoring Formula
score(D,Q) = Σ IDF(qᵢ) · [ f(qᵢ,D)·(k₁+1) / f(qᵢ,D) + k₁·(1 - b + b·|D|/avgdl) ]
k₁ ≈ 1.2–2.0 (term frequency saturation) · b ≈ 0.75 (length normalisation) · avgdl = average document length

BM25 is fast, explainable, and works well when users know the exact terminology. It's the backbone of Elasticsearch, Solr, and Lucene. The problem: it is completely blind to meaning.

Real Failure Examples

# Query: "heart attack" # BM25 misses: "myocardial infarction" (medical term for same thing) # BM25 misses: "cardiac arrest symptoms" # BM25 misses: "chest pain treatment options" # Query: "cheap flights to Mumbai" # BM25 misses: "budget airline tickets to Bombay" # BM25 misses: "low cost carriers Mumbai route" # BM25 misses: "affordable travel options western India" # Query: "fix broken phone screen" # BM25 misses: "cracked display repair guide" # BM25 misses: "LCD panel replacement tutorial" # BM25 misses: "shattered glass smartphone fix"

BM25 is not "bad" — it is excellent at what it does. The problem is that most real-world queries involve intent and meaning, not just exact vocabulary. A user searching for cardiac symptoms doesn't know the clinical term. A traveller doesn't know that "budget airline" and "cheap flight" should map to the same results.

02How Semantic Search Works

The Distributional Hypothesis

The theoretical foundation of semantic search comes from linguistics: "a word is characterised by the company it keeps" (Firth, 1957). Words that appear in similar contexts have similar meanings. "Dog" and "puppy" appear near words like "bark", "leash", "walk" — so they end up with similar vector representations.

This is why embedding models work. They were trained on billions of sentences, learning that "broken screen" and "cracked display" appear in similar contexts — so their embeddings end up close in vector space.

The Geometry of Meaning

When you embed a large corpus, meaning organises itself spatially. Words cluster by topic. Directions in the space carry semantic meaning. The famous example:

# Vector arithmetic preserves meaning king - man + woman ≈ queen Paris - France + Italy ≈ Rome doctor - hospital + school ≈ teacher # This means "gender" is a direction in vector space # "capital city" is a direction in vector space # Meaning has geometry

What the dimensions actually represent — nobody fully knows. The model learns hundreds or thousands of dimensions, and each encodes some latent concept (gender, formality, topic, sentiment...) but they're not labelled or interpretable. This is a known limitation worth mentioning in interviews.

The 3-Step Mental Model

Semantic search has three core steps — everything else is optimisation on top of this:

# Step 1: Embed everything with the same model query_vector = embed("broken phone screen") document_vectors = [embed(doc) for doc in documents] # Step 2: Find nearest neighbours by cosine similarity scores = cosine_similarity(query_vector, document_vectors) # Step 3: Return top-K highest scoring documents top_k = sorted(zip(scores, documents), reverse=True)[:5]

03The Full Pipeline — Offline vs Online

A production semantic search system has two distinct phases. Understanding the split is critical for system design interviews.

Offline Phase — Indexing

This runs once (and again whenever documents change). It's a batch process, latency-insensitive.

indexing pipeline from sentence_transformers import SentenceTransformer import chromadb model = SentenceTransformer('all-MiniLM-L6-v2') client = chromadb.Client() collection = client.create_collection("docs") documents = [ "cracked display repair guide", "fix shattered LCD panel", "mobile screen replacement tutorial", "banana bread recipe", ] # Embed in batches (efficient for large corpora) embeddings = model.encode(documents, batch_size=32, show_progress_bar=True) # Store vectors + metadata in vector DB collection.add( documents=documents, embeddings=embeddings.tolist(), ids=[f"doc_{i}" for i in range(len(documents))], metadatas=[{"source": "support_kb"} for _ in documents] )

Online Phase — Query Time

This runs on every user request. Must be fast — typically under 100ms end-to-end.

query pipeline def semantic_search(query: str, top_k: int = 5): # 1. Embed the query (MUST use the same model as indexing) query_embedding = model.encode([query]) # 2. ANN search in vector DB results = collection.query( query_embeddings=query_embedding.tolist(), n_results=top_k, include=["documents", "distances", "metadatas"] ) # 3. Return ranked results with scores return zip(results['documents'][0], results['distances'][0]) # Usage for doc, score in semantic_search("broken phone screen"): print(f"Score: {score:.3f} | {doc}") # Score: 0.847 | cracked display repair guide # Score: 0.821 | fix shattered LCD panel # Score: 0.794 | mobile screen replacement tutorial # Score: 0.142 | banana bread recipe

Critical: query and documents MUST use the same embedding model. If you indexed with all-MiniLM-L6-v2 and query with text-embedding-3-small, the vectors live in completely different spaces — results will be garbage. If you change models, you must re-embed the entire corpus.

04Approximate Nearest Neighbour Search

Exact nearest neighbour search scans every vector in the database and computes the similarity. At 10 million documents with 768-dimensional vectors, that's 10M × 768 multiplications per query — far too slow for production.

~5ms

HNSW — Hierarchical Navigable Small World

HNSW is the most commonly used ANN algorithm. The intuition: instead of scanning all vectors, build a layered graph where each node connects to its nearest neighbours. Search navigates the graph greedily — starting at the top (sparse) layer and descending to the bottom (dense) layer.

# Key HNSW parameters you'll be asked about: # M = number of connections per node (default 16) # Higher M → better recall, more memory # ef_construction = search depth during build (default 200) # Higher ef_construction → better index quality, slower build # ef_search = search depth at query time # Higher ef_search → better recall, slower query # In ChromaDB, HNSW is used by default # In Pinecone, HNSW and IVF variants are available # In pgvector, choose between HNSW and IVF

The key tradeoff: ANN trades a small recall loss (~5%) for massive speed gains (100–1000x faster than exact search). In practice, the 5% you miss are usually lower-quality results anyway. For interviews, say: "We use ANN because exact search doesn't scale. HNSW gives us sub-10ms latency at millions of vectors with ~95% recall."

05Scoring & Ranking in Detail

Cosine Similarity — Why Angle, Not Distance

Two sentences can differ in length ("I love dogs" vs "I have always genuinely loved dogs my entire life") but mean the same thing. Euclidean distance would make the longer sentence "farther" even if the meaning is identical. Cosine similarity measures the angle between vectors — length-independent, meaning-sensitive.

Cosine Similarity
cos(θ) = (A · B) / (‖A‖ × ‖B‖)
1.0 = same direction (identical meaning) · 0.0 = perpendicular (unrelated) · −1.0 = opposite direction

Dot Product vs Cosine — When They're Equivalent

When vectors are L2-normalised (magnitude = 1), dot product and cosine similarity give identical results. Most embedding models output normalised vectors by default — so in practice, dot product is often used because it's faster to compute.

import numpy as np # Verify normalisation embedding = model.encode("hello world") print(np.linalg.norm(embedding)) # → ~1.0 if normalised # If normalised, these are equivalent: cosine_sim = np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)) dot_product = np.dot(a, b) # faster, same result

Raw cosine similarity scores are not probabilities. A score of 0.85 doesn't mean "85% chance this is the right document". Never present cosine scores as confidence values to users or downstream systems without calibration.

06Hybrid Search — Best of Both Worlds

Pure semantic search fails on exact string matches. A user querying "error code ORA-12541" or "iPhone 15 Pro Max" needs exact lexical match — not nearest semantic neighbour. Pure keyword search misses intent and paraphrasing. Production systems almost always use both.

Reciprocal Rank Fusion (RRF)

RRF is the standard algorithm for merging two ranked lists. Instead of trying to normalise and combine raw scores (which are on different scales), it uses rank positions. The formula:

Reciprocal Rank Fusion
RRF_score(d) = Σ 1 / (k + rank(d))
k = 60 (default constant, reduces impact of top ranks) · rank(d) = document's position in each ranked list · sum across all rankers (BM25, semantic, etc.)
hybrid search with RRF from rank_bm25 import BM25Okapi from sentence_transformers import SentenceTransformer from sklearn.metrics.pairwise import cosine_similarity def reciprocal_rank_fusion(rankings: list[list], k=60): scores = {} for ranked_list in rankings: for rank, doc_id in enumerate(ranked_list): scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank + 1) return sorted(scores.items(), key=lambda x: x[1], reverse=True) def hybrid_search(query, documents, top_k=5): # BM25 ranking tokenised = [doc.split() for doc in documents] bm25 = BM25Okapi(tokenised) bm25_scores = bm25.get_scores(query.split()) bm25_ranking = list(np.argsort(bm25_scores)[::-1]) # Semantic ranking model = SentenceTransformer('all-MiniLM-L6-v2') doc_embs = model.encode(documents) q_emb = model.encode([query]) sem_scores = cosine_similarity(q_emb, doc_embs)[0] sem_ranking = list(np.argsort(sem_scores)[::-1]) # Fuse with RRF fused = reciprocal_rank_fusion([bm25_ranking, sem_ranking]) return [(documents[idx], score) for idx, score in fused[:top_k]]

07Reranking — The Two-Stage Pattern

ANN retrieval is fast but approximate. For high-stakes applications, you do a two-stage retrieval: retrieve a large candidate set quickly (bi-encoder), then rerank precisely (cross-encoder).

Bi-encoders vs Cross-encoders

Bi-encoders embed query and document independently — fast, scalable, used for retrieval. The limitation: they don't see the query and document together, so they miss fine-grained relevance signals. Cross-encoders take the query and document as a pair and produce a single relevance score — much more accurate but O(n) at query time, so only used on a small candidate set.

two-stage reranking from sentence_transformers import CrossEncoder # Stage 1: fast bi-encoder retrieval (top 50) candidates = semantic_search(query, top_k=50) # Stage 2: precise cross-encoder reranking reranker = CrossEncoder('cross-encoder/ms-marco-MiniLM-L-6-v2') pairs = [[query, doc] for doc, _ in candidates] scores = reranker.predict(pairs) # Return top 5 of the reranked 50 reranked = sorted(zip(candidates, scores), key=lambda x: x[1], reverse=True) final_top5 = [doc for (doc, _), _ in reranked[:5]]

The two-stage pattern — retrieve 50, rerank to 5 — is the standard production pattern for any serious retrieval system. Cohere Rerank and BGE Reranker are drop-in cross-encoder options you can mention in interviews.

08When Semantic Search Fails

Knowing when a technique fails is as important as knowing how it works. Interviewers specifically test this.

Exact String Matching Needs

Serial numbers ("ORA-12541"), order IDs, error codes, product SKUs, version numbers — semantic search is wrong for these. The nearest semantic neighbour of "ORA-12541" might be "ORA-12542" which is a completely different error. Use BM25 or direct lookup instead.

Out-of-Domain Vocabulary

An embedding model trained on general web text may not understand highly specialised vocabulary — legal Latin, medical terminology, proprietary internal jargon. Queries containing these terms produce embeddings that are effectively noise. Solution: fine-tune the embedding model on domain-specific data, or use a domain-specific model.

Very Short Queries

A one-word query like "Python" has no context — is the user asking about the language, the snake, or the Monty Python comedy group? The embedding of a single ambiguous word is a poor query vector. BM25 may actually perform better here because it can match the exact word across all contexts.

Stale Embeddings

If a document is updated but its embedding is not re-indexed, search returns the old version's semantics. This is a silent failure — no error, just wrong results. Production systems need a pipeline that detects document changes and triggers re-embedding.

Diagnosing Poor Retrieval

# Step 1: Inspect raw scores — are they all low? # If every result scores < 0.5, the query is likely out-of-domain # Step 2: Visualise embeddings with UMAP import umap reducer = umap.UMAP(n_components=2) reduced = reducer.fit_transform(embeddings) # Plot: does your query cluster near the right documents? # Step 3: Check chunk quality # Bad chunks (too small, cut mid-sentence) produce bad embeddings # Print your chunks and manually verify they're coherent # Step 4: Evaluate with RAGAS (Day 10) # context_precision, context_recall tell you if retrieval is the problem

⚡ Cheat Sheet

One-page interview revision — all key terms, formulas, and 3-sentence answers ready to go.

Get the cheat sheet →

Core concepts

  • BM25 matches exact tokens — fast, explainable, blind to meaning
  • Semantic search embeds query and docs, finds nearest vectors
  • Query and docs MUST use the same embedding model
  • ANN search (HNSW) enables sub-10ms retrieval at millions of vectors
  • Cosine similarity measures angle between vectors, not magnitude

Production patterns

  • Offline: chunk → embed → store in vector DB with metadata
  • Online: embed query → ANN search → top-K → optional rerank
  • Hybrid = BM25 + semantic merged with RRF (k=60)
  • Two-stage: retrieve 50 with bi-encoder, rerank to 5 with cross-encoder
  • Changing models requires full re-embedding of corpus

When to use what

  • Pure BM25: structured data, exact codes, known vocabulary
  • Pure semantic: natural language, intent-driven, multilingual
  • Hybrid: most production systems — best of both
  • Add reranking: high-stakes results where top-5 accuracy matters

Failure modes

  • Exact strings (serial numbers, error codes) → use BM25 instead
  • Out-of-domain vocab → fine-tune or use domain model
  • Very short queries → BM25 may outperform
  • Stale embeddings → need re-indexing pipeline on doc update
  • Low scores across all results → likely out-of-domain query

🎯 Interview Questions — with answers

Click each question to expand the answer. These are the questions most likely to come up for IC4-level AI engineering roles.

Q1 Walk me through what happens step by step when a user submits a query in a semantic search system. +
The query is passed through the same embedding model used to index the documents, producing a query vector. This vector is then sent to the vector database, which uses an ANN algorithm like HNSW to find the top-K most similar document vectors by cosine similarity. The corresponding documents are returned, optionally passed through a cross-encoder reranker for precision, and the final ranked list is returned to the user or passed to an LLM for generation. The critical constraint is that the query and all documents must be embedded with the same model — different models live in incompatible vector spaces.
Q2 Why might you combine keyword search and semantic search in production? +
Pure semantic search fails on exact string matches — product codes, error messages, proper nouns, version numbers — because the nearest semantic neighbour of "ORA-12541" might be a completely different error code. Pure keyword search misses synonyms, paraphrases, and intent. Hybrid search combines both: BM25 handles exact matching and structured data, semantic search handles natural language and meaning. Most production systems I've seen (and built) use hybrid by default because it outperforms either approach alone across the broadest range of queries.
Q3 What is Reciprocal Rank Fusion and why k=60? +
RRF merges multiple ranked lists by assigning each document a score of 1/(k + rank) in each list, then summing across lists. The key insight is that it uses rank positions, not raw scores — so you never need to normalise BM25 scores and cosine similarities onto the same scale. The constant k=60 was found empirically to reduce the outsized influence of rank-1 results while still rewarding consistently high-ranked documents across multiple lists. It's robust, parameter-free in practice, and works well even when one ranker has significantly more results than the other.
Q4 What are the main failure modes of semantic search? +
Four main failure modes. First, exact string matching needs — serial numbers, error codes, IDs — where you need lexical match, not semantic neighbour. Second, out-of-domain vocabulary — a model trained on web text doesn't understand medical Latin or proprietary jargon, producing embeddings that are effectively noise. Third, very short or ambiguous queries — a single word like "Python" has no context, so the embedding is ambiguous and retrieval is poor. Fourth, stale embeddings — documents updated after indexing return results based on old content, a silent failure with no error signal. For each of these, the fix is different: hybrid search for exact matching, domain fine-tuning for vocabulary gaps, query expansion for short queries, and a re-indexing pipeline for freshness.
Q5 How does ANN search differ from exact nearest neighbour, and why does it matter? +
Exact nearest neighbour search computes similarity against every vector in the database — O(n·d) per query. At 10 million vectors with 768 dimensions, that's 7.6 billion multiplications per query, which is far too slow for interactive latency. ANN algorithms like HNSW build a graph structure that allows greedy navigation to approximate neighbours in O(log n), trading ~5% recall for 100-1000x speed improvement. In practice, the 5% you miss are usually lower-quality results anyway. The key parameters to know: M (connections per node), ef_construction (build quality), and ef_search (query-time recall vs speed tradeoff).
Q6 If semantic search quality is poor, what would you investigate first? +
I'd start with chunk quality — bad chunking (too small, split mid-sentence, no context overlap) is the single most common cause of poor retrieval, and it's invisible unless you print and manually inspect your chunks. Second, I'd look at raw cosine scores — if everything is below 0.5, the query is likely out-of-domain for the embedding model. Third, I'd check whether the query and index are using the same model — a mismatch produces garbage results silently. Finally, I'd run RAGAS evaluation metrics — context precision and context recall — to precisely quantify where in the pipeline retrieval is losing information. Most poor retrieval issues trace back to one of: bad chunks, wrong model, or out-of-domain vocabulary.
